Task Overview & Prompt Engineering Strategy

How the system prompt and tool schema were designed to drive the agent loop

Assignment: Design an Agentic Workflow that processes multimodal inputs (an industrial electrical panel wiring diagram image), extracts structured data via a planned tool-call loop, and produces a documented extraction pipeline. Deliverable: a dynamic interactive report with zoomable visualizations.

🔧 System Prompt Design

# System prompt (abridged) You are an expert electrical engineering AI. You have access to the following tools: extract_diagram_metadata # Step 1 — ground truth anchor extract_component_list # Step 2 — structured JSON schema extract_wire_specifications # Step 3 — single responsibility build_bom # Step 4 — context chaining audit_protection_devices # Step 5 — safety verification generate_power_flow_graph # Step 6 — graph as JSON compile_final_report # Step 7 — forced termination RULES: - Call tools sequentially; each step builds on prior results. - Never skip steps. - Return structured JSON from every tool call. - Use end_turn ONLY after Step 7.

Sequential ordering + JSON output schema are the two most impactful prompt design decisions — see §2 for detailed rationale.

📌 Multimodal Input Strategy

The user turn passes the wiring diagram as a base64 encoded image alongside a task description. The model reasons over the visual content and emits structured tool calls — never raw image descriptions:

messages: [{ "role": "user", "content": [ { "type": "image", "source": { "type": "base64", "media_type": "image/jpeg", "data": <diagram_b64> } }, { "type": "text", "text": "Extract all structured data from this wiring diagram using the available tools." } ] }]

Prompt Engineering Rationale

Why each design decision was made — the documented thought process behind every step

Why does the order of tool calls matter?
Free-form ReAct loops (where the model decides its own order) frequently produce partial extractions because the model terminates early once it believes the task is "done enough." Enforced sequential ordering guarantees completeness — every step must execute before end_turn is permitted. This is the single highest-impact prompt engineering decision in this workflow.
Step Tool Prompt Engineering Rationale
Step 1 extract_diagram_metadata Instructing the model to start with metadata grounds every subsequent step in verified context (drawing number, voltage levels, standard). Without this anchor step the model could hallucinate component ratings that contradict the actual diagram header.
Step 2 extract_component_list Requiring JSON output with fixed keys (id, type, rating, zone, wire_spec, line_ref) forces the model to produce machine-readable records rather than free-form prose. Fixed schema also makes downstream tool calls deterministic — build_bom can reliably reference the same field names.
Step 3 extract_wire_specifications Separating wire extraction into its own step prevents the component-list step from becoming overloaded. Cognitive load management in the prompt — one responsibility per tool call — measurably reduces hallucinated wire gauges compared to asking for everything in a single call.
Step 4 build_bom This step uses the output of Steps 2–3 as explicit context injected into the prompt. Chaining structured JSON outputs as inputs forces the model to reconcile any inconsistencies (e.g. a component listed in Step 2 but without a wire spec in Step 3) rather than silently omitting items.
Step 5 audit_protection_devices By the time the agent reaches this step it holds the full component list and wire specs in context. The system prompt explicitly forbids skipping steps, ensuring the audit always has verified input data — this is the key safety benefit of enforced sequential ordering over a free-form ReAct loop.
Step 6 generate_power_flow_graph Asking the model to output a graph as a structured node/edge JSON (not a description) means the visualization layer can render it directly without a second parsing step. This separation of 'data extraction' from 'rendering' is a core agentic design principle demonstrated here.
Step 7 compile_final_report The final step serves as a forcing function: the model cannot emit end_turn until it has called compile_final_report. This prevents premature termination — a common failure mode in free-form agent loops where the model decides it is 'done' after only partial extraction.
